# Hyperparameter Extraction and Classification

You are a coding agent tasked with extracting and classifying tunable hyperparameters from a Hydra-based ML project. Your goal is to identify which parameters are worth tuning, classify their types, and define reasonable search spaces.

## Input

You should already have:
1. The project's `HPARAM.md` (generated by `get_hparam_structure.md` prompt)
2. Access to the project's Hydra config files
3. Optionally, the training command used by the user

## Procedure

### Step 1: Enumerate All Config Parameters

1. Read the root config file (e.g., `conf/config.yaml`) and follow its `defaults:` list.
2. For each config group, read the active YAML file and list every key-value pair.
3. Use `python train.py --cfg job` (or the project's entry point) to dump the full resolved config.
4. Flatten the hierarchical config into dot-separated paths:
   ```
   optimizer.lr = 0.001
   optimizer.weight_decay = 0.0001
   model.hidden_size = 256
   model.num_layers = 4
   model.dropout = 0.1
   trainer.max_epochs = 100
   trainer.batch_size = 32
   trainer.gradient_clip_val = 1.0
   dataset.path = /data/cifar10
   dataset.num_workers = 4
   output_dir = ./outputs
   seed = 42
   debug = false
   ```

### Step 2: Classify Each Parameter

Assign each parameter to one of these categories:

#### Category A: Core Optimization Hyperparameters (HIGH priority)
Parameters that directly control the optimization process. These are almost always worth tuning.

| Parameter Pattern | Type | Scale | Typical Range | Notes |
|---|---|---|---|---|
| `*.lr`, `*.learning_rate` | float | log | [1e-6, 1e-1] | Most impactful hparam |
| `*.weight_decay` | float | log | [1e-6, 1e-1] | Regularization strength |
| `*.batch_size` | int | linear (powers of 2) | [8, 512] | Constrained by GPU memory |
| `*.warmup_steps`, `*.warmup_epochs` | int | linear | [0, 10% of total] | Often critical for transformers |
| `*.dropout` | float | linear | [0.0, 0.5] | Regularization |
| `*.label_smoothing` | float | linear | [0.0, 0.2] | Classification regularization |

#### Category B: Model Architecture Hyperparameters (MEDIUM priority)
Parameters that define model structure. Tuning is valuable but more expensive.

| Parameter Pattern | Type | Scale | Notes |
|---|---|---|---|
| `*.hidden_size`, `*.d_model` | int | linear (powers of 2) | Capacity control |
| `*.num_layers`, `*.depth` | int | linear | Depth vs. speed tradeoff |
| `*.num_heads`, `*.nhead` | int | linear (factors of d_model) | Attention heads |
| `*.intermediate_size`, `*.d_ff` | int | linear | FFN width |
| `*.embed_dim`, `*.embedding_dim` | int | linear | Embedding dimension |

#### Category C: Training Strategy Hyperparameters (MEDIUM priority)
Parameters controlling training procedure but not the optimizer directly.

| Parameter Pattern | Type | Scale | Notes |
|---|---|---|---|
| `*.max_epochs`, `*.num_epochs` | int | linear | Training duration |
| `*.gradient_clip_val`, `*.max_grad_norm` | float | linear | [0.1, 10.0] |
| `*.accumulate_grad_batches` | int | linear (powers of 2) | Effective batch size |
| `*.precision`, `*.amp` | categorical | â€” | {16, 32, "bf16"} |
| `*.scheduler`, `*.lr_scheduler` | categorical | â€” | {cosine, linear, step} |
| `*.early_stopping_patience` | int | linear | Early stopping |

#### Category D: Non-Tunable Parameters (SKIP)
Parameters that should NOT be automatically tuned:

| Parameter Pattern | Reason |
|---|---|
| `*.path`, `*.dir`, `*.root`, `*_dir` | Data/output paths |
| `*.num_workers`, `*.pin_memory` | Dataloader efficiency (hardware-dependent) |
| `seed`, `*.seed` | Reproducibility control |
| `debug`, `*.debug`, `*.verbose` | Debug flags |
| `output_dir`, `log_dir`, `save_dir` | Output management |
| `*.resume`, `*.checkpoint` | Training resumption |
| `wandb.*`, `logger.*`, `logging.*` | Logging config |
| `*.device`, `*.gpus`, `*.accelerator` | Hardware config |

### Step 3: Infer Search Spaces

For each tunable parameter, determine:

1. **Type**: `float`, `int`, `bool`, or `categorical`
2. **Scale**: `linear` or `log` (use `log` for learning rates, weight decay, etc.)
3. **Range**: Based on the current value and domain knowledge:
   - For `float` with `log` scale: typically [current / 10, current * 10]
   - For `float` with `linear` scale: typically [current * 0.2, current * 5] or a domain-specific range
   - For `int`: typically [current // 2, current * 2] or domain-specific
   - For `categorical`: list all reasonable choices
4. **Priority**: `high`, `medium`, `low`, or `skip`

### Step 4: Handle Command-Line Context

If the user provides a specific training command like:
```bash
python train.py optimizer=adam optimizer.lr=0.001 model=resnet model.dropout=0.1 trainer.max_epochs=50
```

Then:
1. The overridden parameters are likely the ones the user considers important â€” prioritize them.
2. Parameters NOT in the override but in the config are still candidates but may have lower priority.
3. The config group choices (e.g., `optimizer=adam`) indicate which optimizer-specific params are relevant.

### Step 5: Output Format

Generate a `hparam_space.yaml` file with the following structure:

```yaml

# Project: <project_name>
# Generated from: <config_files_inspected>
# Training command: <command_if_provided>

    type: float

    category: core_optimization

  - name: optimizer.weight_decay
    type: float
    low: 1.0e-6
    high: 1.0e-1

  # --- Category B: Model Architecture (MEDIUM priority) ---
  - name: model.dropout

    low: 0.0
    high: 0.5
    log: false
    priority: medium
    category: model_architecture

  # --- Category C: Training Strategy (MEDIUM priority) ---
  - name: trainer.gradient_clip_val
    type: float
    low: 0.1
    high: 10.0
    log: true
    priority: medium

    current_value: 1.0

  # --- Skipped parameters ---
  # dataset.path: /data/cifar10 (non-tunable: data path)
  # output_dir: ./outputs (non-tunable: output path)

  # debug: false (non-tunable: debug flag)

summary:
  total_params_found: 25
  tunable_high: 3
  tunable_medium: 5
  tunable_low: 4
  skipped: 13
```

## Important Guidelines

1. **Be conservative with ranges**: Extreme values can waste compute. Start with ranges centered around the current value unless diagnostics suggest otherwise.
2. **Respect parameter interactions**: If batch_size is tuned, consider adjusting learning rate proportionally (linear scaling rule).
3. **Consider compute budget**: If the user has limited GPU time, recommend tuning only high-priority parameters first.
4. **Flag potential issues**: If a parameter has an unusual value (e.g., lr=1.0 for Adam), flag it as potentially misconfigured.
5. **Preserve non-tunable params**: Never include dataset paths, output directories, or debug flags in the search space.
